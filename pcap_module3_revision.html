Παγίδες & Προσοχή

Οι λεπτομέρειες που τρώνε πόντους από οριακούς μαθητές. Αυτές χωρίζουν το «το ξέρω περίπου» από το «το πετυχαίνω σίγουρα».

📌 Τα strings είναι immutable — αλλά το slicing είναι ασφαλές

s[0] = "x" ή del s[0] → TypeError· s.append(...) → AttributeError. Το slicing όμως φτιάχνει νέο string και δεν πειράζει το αρχικό.

📌 Slicing εκτός ορίων = ΟΚ · indexing εκτός ορίων = error

"string"[2:100] → "ring" (κανένα error). Αλλά "string"[9] → IndexError. Το slice «κόβει» στα όρια· το index απαιτεί υπαρκτή θέση.

📌 .find() επιστρέφει -1 · .index() πετάει εξαίρεση

Αν δεν βρεθεί το substring: .find() → -1 (ασφαλές), ενώ .index() → ValueError. Μην τα μπερδέψεις.

📌 ord / chr: πεζά > κεφαλαία κατά 32

ord('a')==97, ord('A')==65, διαφορά 32 (= το code point του space). Άρα ord('x') - ord('X') == ord(' ') είναι True. Τα πεζά έχουν μεγαλύτερα code points.

📌 Το κενό string «υπάρχει» παντού

"" in s είναι πάντα True, ακόμη κι αν το s είναι κενό. Προσοχή: x not in s δεν είναι το ίδιο με not x in s ως προς προτεραιότητα ανάγνωσης — αλλά δίνουν το ίδιο αποτέλεσμα.

📌 str + int → TypeError · str * int → επανάληψη

Δεν μπορείς να ενώσεις string με int ("a" + 2 → TypeError). Όμως "a" * 3 → "aaa". Συχνή παγίδα σε code-trace με for c in s + 2.

📌 .split() vs .join()

.split() χωρίς όρισμα σπάει στα κενά. Το .join() θέλει λίστα με μόνο strings — αν υπάρχει int μέσα → TypeError. Σύνταξη: sep.join(list) (ο separator καλεί το join).

📌 Οι μέθοδοι δεν αλλάζουν το αρχικό

Επειδή τα strings είναι immutable, s.upper(), s.replace(...), s.strip() κ.λπ. επιστρέφουν νέο string. Αν δεν το αναθέσεις (s = s.upper()), το αρχικό μένει ίδιο.

📌 is-μέθοδοι: τι μετράει ως «ναι»

isalnum = γράμματα ή ψηφία· isalpha = μόνο γράμματα· isdigit = μόνο ψηφία. Π.χ. "R2D2".isalpha() → False (έχει ψηφία), "R2D2".isalnum() → True.
